Stage and commit changes following conventional commit format.

## Arguments

Optional arguments the user may provide:
- **type**: Commit type (e.g., `feat`, `fix`, `chore`). If not provided, infer from the changes.
- **scope**: Module scope (e.g., `workspace`, `auth`). If not provided, infer from changed files.
- **message**: Description text. If not provided, generate from the diff.

## Steps

### 1. Check state

```bash
git diff --cached --stat && echo "---" && git diff --stat && echo "---" && git ls-files --others --exclude-standard
```

If nothing to commit, tell the user and stop.

### 2. Stage changes

If nothing is staged, run `git add -A` to stage everything. If something is already staged, commit what's staged as-is.

### 3. Build the commit message

Get the diff:
```bash
git diff --cached
```

**Type** — infer from changes if not provided:
- New features → `feat`
- Bug fixes → `fix`
- Tests only → `test`
- Config/deps/build → `chore`
- Restructure without behavior change → `refactor`
- Docs only → `docs`
- Style/formatting → `style`
- Performance → `perf`

**Scope** — infer from changed files (e.g., `workspace`, `auth`, `api`, `hooks`, `components`). Omit if changes span many unrelated areas. Lowercase + hyphens only.

**Description** — concise, lowercase, no period. Subject line under 72 characters.

Format: `type(scope): description`

### 4. Commit

```bash
git commit -m "{commit_message}"
```

If the commit fails, show the error and help fix it.

### 5. Output

Print:
```
✔ {short_hash} {commit_message}
```

Then ask if they want to push.

**NEVER add AI co-author, "generated by", "assisted by", or any AI attribution to commit messages.**
